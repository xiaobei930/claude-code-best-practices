# 通用编码标准

本规则适用于所有语言和项目，定义核心编码原则。

## 代码质量原则

1. **可读性优先** — 清晰命名胜过注释，简单代码胜过聪明代码，一致风格胜过个人偏好
2. **KISS** — 选择最简单的可行方案，不做过早优化
3. **DRY** — 提取公共逻辑为函数，但 3 次重复再抽象
4. **YAGNI** — 不构建未来可能需要的功能，先简单实现，需要时再重构

## 命名规范

- **变量**: 描述性名称（`userSearchQuery` 非 `q`，`isAuthenticated` 非 `flag`）
- **函数**: 动词+名词（`fetchUserData`、`calculateTotal`、`isValidEmail`）
- **布尔**: is/has/can/should 前缀（`isLoading`、`hasPermission`、`canEdit`）

## 不可变性原则

永远不要直接修改对象或数组。使用展开运算符创建新对象、使用 filter/map 创建新数组。

## 错误处理

- 完整的 try/catch，不忽略错误
- 自定义错误类区分错误类型
- 异步操作使用 `Promise.all` 并行，`Promise.allSettled` 处理部分失败

## 代码组织

| 指标     | 推荐       | 上限   |
| -------- | ---------- | ------ |
| 文件大小 | 200-400 行 | 800 行 |
| 函数大小 | 20-30 行   | 50 行  |
| 嵌套深度 | —          | 3 层   |

- 使用早返回（guard clause）减少嵌套
- 单一职责原则

## 注释规范

- 解释"为什么"，而非"是什么"
- 公共 API 使用 JSDoc/docstring
- 好的代码是自解释的——需要大量注释通常意味着需要重构

## 代码异味

- 超过 50 行的函数 → 拆分
- 魔法数字 → 命名常量
- 上帝类/函数 → 按职责拆分
- 相似代码 3 次以上 → 提取公共函数

---

> **记住**：好的代码是自解释的。如果需要大量注释来解释代码，通常意味着代码需要重构。
